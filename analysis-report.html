Executive Summary High-level findings and overall score at a glance

Trailblazers' Vault is a Next.js web application designed to help tabletop RPG players and Game Masters manage shared party inventories. The application features a warm, medieval aesthetic, a drag-and-drop inventory system, Firebase-backed authentication, and real-time campaign collaboration.

The overall UX is above average for a semester project, with notable strengths in visual design, domain-specific language, and the availability of a dedicated Help/Guides section. However, several medium-to-high severity issues were identified in error prevention, error recovery, and interaction efficiency that should be addressed before the application is considered production-ready for a wider audience.

Heuristic-by-Heuristic Evaluation Scores (1 = very poor → 5 = excellent), risk labels, observed issues and recommendations

1
Visibility of System Status
3 / 5 Medium Risk
Observed Issues
  • No visible loading spinner or skeleton screen during Firebase data fetches; the page can appear blank briefly. UX
  • No real-time sync indicator (e.g., "Syncing…" / "All changes saved") to reassure users that their shared vault is up to date. UX
  • Transfer request expiry is time-based but the countdown is not displayed to the sender or recipient. UX
Positive Observations
  • "You're already signed in!" toast notification is immediate and clear.
  • Animated scroll indicator ("Scroll for more") on the landing page effectively communicates more content below.
Recommendations
  • Add a skeleton loading state or subtle spinner for all Firebase fetch operations.
  • Show a persistent sync-status chip in the vault header (e.g., green dot + "Saved" or amber "Saving…").
  • Display remaining time on pending transfer requests to reduce user uncertainty.
2
Match Between System and the Real World
4 / 5 Low Risk
Observed Issues
  • The term attunement on item cards is meaningful only to D&D players; non-TTRPG users may not understand it without a tooltip. UX
  • "Campaign ID" entry modal uses an abstract alphanumeric code that has no real-world parallel; the label could be clearer (e.g., "Invite Code"). UX
Positive Observations
  • Vocabulary ("vault", "loot", "party", "Game Master") is highly authentic to the target TTRPG audience.
  • Icons (Package, Archive, BookOpen, GripVertical) map naturally to inventory management concepts.
  • Item categories (weapon, armor, potion) mirror D&D source books.
Recommendations
  • Add a tooltip or "?" icon next to attunement with a one-sentence explanation.
  • Rename "Campaign ID" to "Invite Code" and add a brief description below the input field.
3
User Control and Freedom
3 / 5 Medium Risk
Observed Issues
  • Drag-and-drop item moves appear irreversible within the session; there is no undo (Ctrl+Z) or visible history. UX
  • No "Cancel" button on the Add Item or Item Details modals if opened accidentally on mobile (the escape route requires clicking the overlay). UX
  • Vault/campaign deletion flow (if it exists) lacks an explicit "Leave Campaign" or "Delete Vault" cancel path. UX
Positive Observations
  • Custom not-found.tsx page exists, preventing dead-end navigation.
  • Navigation bar is persistent, providing a clear route back to the home and vaults pages.
Recommendations
  • Implement a basic action history (last 5 moves) with an "Undo last action" button in the vault toolbar.
  • Ensure all modals have a clearly labelled ✕ close button accessible via both mouse and keyboard (Escape key).
4
Consistency and Standards
3 / 5 Medium Risk
Observed Issues
  • Several hero buttons are implemented as <button> nested inside <a> tags, which is invalid HTML and produces unexpected behaviour with assistive technologies. A11Y
  • Multiple // @ts-ignore comments in the vault page suppress TypeScript errors in the drag-and-drop library integration, indicating type-safety inconsistency. UX
  • Button border-width varies between 2px and 4px on the same page without a clear design rationale. UX
Positive Observations
  • Colour palette (earthy browns and deep reds) is applied consistently throughout all pages.
  • Typography hierarchy (Archivo Black for headings, Averia Gruesa Libre for body) is used uniformly.
Recommendations
  • Replace <a><button>…</button></a> patterns with a single semantic element (<a> or <button> with onClick).
  • Define a Tailwind component class for primary, secondary, and ghost buttons to enforce consistent border widths and padding.
  • Resolve @ts-ignore comments by using proper type declarations for the DnD libraries.
5
Error Prevention
2 / 5 High Risk
Observed Issues
  • No confirmation dialog before deleting an item or vault — destructive actions execute immediately. UX
  • Authentication state is stored in localStorage (key trailblazers-auth) in addition to Firebase; stale local state could mismatch Firebase auth, causing incorrect access to protected routes. UX
  • The sign-up password field accepts any 6+ character string without a complexity indicator; weak passwords can be set silently. UX
  • Transfer requests can expire silently; neither party is notified when an expired transfer is cleaned up. UX
Positive Observations
  • Client-side email format validation (EMAIL_RE) is applied on sign-up before hitting Firebase.
  • Guided tutorial steps during sign-up progressively expose form fields, reducing accidental input errors.
Recommendations
  • Add a confirmation modal ("Are you sure? This cannot be undone.") for all destructive actions.
  • Remove the duplicated localStorage auth store; rely solely on Firebase's onAuthStateChanged for authentication state.
  • Add a password strength indicator with colour coding (weak / fair / strong) on the sign-up form.
  • Send an in-app notification or toast when a pending transfer expires.
6
Recognition Rather than Recall
3 / 5 Medium Risk
Observed Issues
  • The drag-and-drop interaction is not visually indicated — there are no grab handles (despite GripVertical being imported on the landing page, it is absent in the actual inventory grid). UX
  • Inventory categories (weapon, armor, potion, etc.) are displayed as text labels without category icons in the item card, requiring the user to read rather than recognise. UX
  • Campaign ID is an abstract code; there is no copy-to-clipboard affordance visible in the UI, so users may not know they can share it. UX
Positive Observations
  • All CTA buttons use icon + text labels (Package, Zap, HelpCircle) — no icon-only buttons on critical actions.
  • The tutorial step system during sign-up walks the user through every required field, eliminating the need to recall what information is needed.
Recommendations
  • Add visible grab-handle icons (e.g., GripVertical) to draggable item rows in the inventory.
  • Add small category icons (⚔ for weapon, 🛡 for armor, 🧪 for potion) alongside text labels in item cards.
  • Display the Campaign/Invite Code prominently with a one-click "Copy Code" button.
7
Flexibility and Efficiency of Use
2 / 5 High Risk
Observed Issues
  • No search or filter functionality on the vault inventory page; users with large inventories must scroll to find items. UX
  • No keyboard shortcuts (e.g., N for New Item, / to focus search) for power users. UX
  • No bulk-select or multi-move functionality; items can only be moved one at a time. UX
  • No way to sort inventory columns (by name, rarity, weight, value). UX
Positive Observations
  • Drag-and-drop is itself an efficient interaction pattern — significantly faster than form-based item management.
  • Template items speed up the "add common item" workflow for returning users.
Recommendations
  • Add a real-time search/filter bar above the inventory list (filter by name, category, rarity).
  • Add column sort controls to the inventory table/grid.
  • Document keyboard shortcuts in the Guides page and implement at least Esc, Enter, and N for the core inventory flow.
8
Aesthetic and Minimalist Design
4 / 5 Low Risk
Observed Issues
  • The landing page hero section presents three equally prominent CTA buttons (Manage Vaults, Quick Start, Need Support), which can cause decision paralysis for first-time visitors. UX
  • The About section on the landing page repeats information already visible in the Features section, adding visual weight without new value. UX
Positive Observations
  • The warm earthy colour palette (burgundy, tan, gold) creates a strong and memorable medieval RPG identity.
  • Subtle backdrop blur on feature cards (backdrop-blur-sm) adds depth without distraction.
  • Generous whitespace and consistent border-radius create a polished, uncluttered layout.
  • Scroll-reveal animations are tasteful and do not impede usability.
Recommendations
  • Elevate the primary CTA ("Manage Vaults") visually — make it larger and more prominent, demoting Support to a text link.
  • Consider consolidating or removing the duplicate About section, or replace it with social proof (player count, vault count).
9
Help Users Recognise, Diagnose, and Recover from Errors
2 / 5 High Risk
Observed Issues
  • Firebase write failures (e.g., network loss during item move or transfer) are not visibly surfaced to the user — there is no error toast or retry prompt for async operations in the vault. UX
  • Rejected transfer requests display a toast but do not explain why the transfer was rejected or what the user should do next. UX
  • Form validation errors in the vault's Add Item modal are not described in the code, suggesting they may be absent or only rely on browser built-ins. UX
Positive Observations
  • Login form correctly sets loginError and signupErrors states, displaying inline validation messages.
  • A dedicated Support and FAQ page is accessible from the main navigation.
Recommendations
  • Wrap all Firebase write operations in try/catch and display a toast with a meaningful message and "Retry" action on failure.
  • When a transfer is rejected, show the recipient's name (or "the other player") and a "Retry transfer" button.
  • Add inline validation to the Add Item modal (e.g., "Name is required", "Value must be a positive number").
10
Help and Documentation
4 / 5 Low Risk
Observed Issues
  • Help is isolated in the Guides and Support pages; there are no contextual tooltips or inline hints within the vault inventory view itself. UX
  • The FAQ page's content depth is not visible from the code; it may not cover the most common user errors (item transfer issues, campaign join failures). UX
Positive Observations
  • A dedicated /guides page with a Quick Start route is accessible from the hero section and navigation.
  • A dedicated /support page with a nested /support/faq structure is available.
  • The sign-up tutorial messages are excellent in-context help for new users.
Recommendations
  • Add a small "?" help button or first-time-user overlay (coach marks) on the vault inventory page to orient new users to the drag-and-drop interface.
  • Surface the top 3 most relevant FAQ answers as a collapsible section directly on the vault page.

Stakeholder Summary Audience-specific takeaways for product owners, developers, and designers

📊 For the Product Owner

The application is well-scoped and addresses a real need in the TTRPG community. The most impactful near-term investments that will reduce user frustration and build trust are:

  • Confirmation dialogs for all delete actions (1–2 hours of development)
  • Persistent sync status in the vault header (2–3 hours)
  • Real-time inventory search (3–5 hours)

These three items alone can raise the UX score from 3.2 to approximately 3.8 / 5.0.

🧑‍💻 For the Development Team

Three technical debt items should be addressed alongside new features:

  • Remove duplicated localStorage auth — rely solely on onAuthStateChanged
  • Replace all <a><button> nesting with a single semantic element
  • Resolve @ts-ignore comments with proper DnD type declarations

Wrapping every Firebase mutation in try/catch with error toasts is a high-value, low-complexity improvement.

🎨 For the UX/Design Team

The visual design is the strongest area of the application — the RPG aesthetic is authentic and cohesive. Priority design tasks are:

  • Add visible drag-handle affordances (GripVertical) to inventory item rows
  • Differentiate the primary CTA ("Manage Vaults") from secondary actions on the landing page
  • Add category icons to item cards for faster visual scanning
  • Create a first-time-user onboarding coach mark overlay for the vault view

♿ Accessibility & Quality

The application has a baseline level of accessibility but should improve in these areas before release:

  • Fix <button> inside <a> (WCAG 4.1.1 Parsing)
  • Ensure all modals trap focus and are dismissible via Escape (WCAG 2.1.1 Keyboard)
  • Verify colour contrast ratios for muted text on the parchment background
  • Add aria-live regions for toast notifications

Mobile Responsiveness
4 / 5 Low Risk

Tailwind's responsive grid classes (sm:grid-cols-3) and the TouchBackend for drag-and-drop demonstrate strong mobile-first thinking. The main gap is testing the drag-and-drop experience on actual touch devices — touch DnD often requires longer press delays and more explicit visual feedback.

Performance
3 / 5 Medium Risk

Next.js with Vercel deployment ensures good baseline performance. However, all Google Fonts are loaded via <link> in the custom <head> in addition to Next.js font optimisation, which risks duplicate requests. Firebase listeners on the vault page (subscribeToCampaign, subscribeToPlayerInventories, etc.) should be audited to confirm all are unsubscribed on component unmount.

Security & Privacy
3 / 5 Medium Risk

Firebase security rules (rules-firebase.md) are a positive sign. The main concern is the duplicated auth state in localStorage — this state is not cryptographically verified and could be trivially forged by a malicious user to gain access to the /vaults route client-side (though server-side Firebase rules should still enforce access).

Accessibility (WCAG 2.1)
2 / 5 High Risk

Beyond the invalid HTML nesting, the drag-and-drop interface has no keyboard alternative, excluding all motor-impaired users from the core workflow. Toast notifications lack aria-live regions. The muted text colour (#5C4A2F on #E8D5B7) should be verified to meet a 4.5:1 contrast ratio for body text.
